# PRD Template

## Title

Short, specific name of the change.

## Background

Why this work matters now.

## Problem

What is broken, missing, confusing, or risky.

## Goal

The outcome this PR should achieve.

## Non-goals

What this PR explicitly will not do.

## Requirements

Concrete requirements the implementation must satisfy.

## Data Changes

New fields, schema changes, normalization changes, migrations, or compatibility notes.

## UI Changes

User-visible changes, labels, inputs, charts, or dashboard changes.

## Backward Compatibility

How existing CSV records, JSON imports, and user workflows remain compatible.

## Acceptance Criteria

Checklist for accepting the work.

## Validation

Commands and realistic scenarios used to verify the work.

## Risks

Known risks, tradeoffs, or areas needing review.

## Future Scope

Useful follow-up work that is outside this PR.
